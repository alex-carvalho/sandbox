# Loki - Scaled Architecture (Microservices)

This directory contains a configuration to run **Grafana Loki** in a microservices (scalable) architecture. The read, write, and backend components are separated to allow independent scaling of each part of the log pipeline.

### Components:

1. **Loki Write (`write`)**:
   * A Loki instance configured with the `-target=write` target.
   * Responsible for receiving logs sent by Alloy, indexing them, and temporarily saving them to the WAL (Write-Ahead Log) before uploading them to object storage.

2. **Loki Read (`read`)**:
   * A Loki instance configured with the `-target=read` target.
   * Responsible for processing log queries (LogQL) coming from Grafana.

3. **Loki Backend (`backend`)**:
   * A Loki instance configured with the `-target=backend` target.
   * Responsible for background tasks such as file compaction (`compactor`), rule evaluation (`ruler`), etc.

4. **Gateway (`gateway`)**:
   * An Nginx server acting as a unified reverse proxy on port `3100`.
   * Automatically routes write requests (`/loki/api/v1/push`) to the **Write** instances, and read/query/tail requests (`/loki/api/.*`) to the **Read** instances.

5. **MinIO (`minio`)**:
   * A local S3-compatible object storage.
   * Used by Loki to store persisted logs in the `loki-data` and `loki-ruler` buckets.

6. **Grafana Alloy (`alloy`)**:
   * A modern log collector from Grafana.
   * Monitors the Docker Socket (`/var/run/docker.sock`) to discover and collect log streams from other running containers.
   * Labels and ships logs to the Gateway with the tenant ID header set to `tenant1`.

7. **Grafana (`grafana`)**:
   * Available on port `3000`.
   * Pre-provisioned with a Loki Data Source pointing to the Gateway (`http://gateway:3100`) and configured with the `X-Scope-OrgID: tenant1` header for basic multi-tenant authentication.

8. **Flog (`flog`)**:
   * A dummy log generator in JSON format for simulation and testing purposes.

## 🚀 Getting Started

### Prerequisites
* Docker and Docker Compose installed.

### Step-by-Step

1. **Start the services:**
   In the `scale` directory, run:
   ```bash
   docker compose up -d
   ```

2. **Verify startup logs:**
   ```bash
   docker compose logs -f
   ```

3. **Access Grafana:**
   * Open your browser at: [http://localhost:3000](http://localhost:3000)
   * Login is configured as Anonymous Admin (no password required).

4. **Query logs:**
   * Go to the **Explore** section (sidebar -> `Explore`).
   * Select the **Loki** datasource.
   * Run a simple log query, for example: `{container="scale-flog-1"}` or just `{container=~".+"}` to view logs generated by flog and other containers.

## ⚖️ Scaling the Infrastructure

One of the main advantages of this architecture is the ease of scaling `write` and `read` nodes individually. 

To scale the setup, run:

```bash
docker-compose up -d --scale write=3 --scale read=2
```

This will spin up:
* **3 replicas** of the `write` service to handle ingestion spikes.
* **2 replicas** of the `read` service to handle multiple concurrent queries.
* **1 Nginx gateway** container that acts as a unified entrypoint and automatically load-balances connections.

### How Scaling Works Under the Hood:
1. **No Host Port Collisions:** The static host port mappings (such as `3101:3100` and `3102:3100`) have been removed from the `read`, `write`, and `backend` services and replaced with internal container `expose`. This prevents port conflicts on the host when launching multiple replicas. Only the Nginx `gateway` (`3100`) and `grafana` (`3000`) services publish ports to the host.
2. **Dynamic Upstream Resolution:** Nginx is configured to use variables for its upstream targets (e.g., `set $upstream_write write; proxy_pass http://$upstream_write:3100...`). This forces Nginx to dynamically query Docker's internal DNS resolver (`127.0.0.11`) on each request (or according to TTL) rather than caching a single IP at startup, ensuring seamless load balancing and compatibility with replica scale-up/down.
